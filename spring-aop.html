Spring AOP

  • by:Mohammed NAIMI
  • date:February 19, 2024

Each application needs to have methods executing, exceptions handled, and security checks performed seamlessly. Each of these is crucial, but they can also clutter your core business logic, making it harder to understand and maintain. This is where Spring AOP steps in, offering a powerful way to separate these cross-cutting concerns and keep your code clean and modular.

Spring AOP demystified :

Joinpoints: These are specific points in your application's execution flow, like method calls or exception throws. Think of them as defined moments where you can inject your custom behavior.
Pointcuts: These act as filters, precisely defining which joinpoints your aspect applies to. They're like targeting mechanisms, ensuring your aspect's influence reaches the relevant areas of your code.
Advice : This is the action your aspect takes at a joinpoint. It could be logging a message before a method call, adding security checks, or gracefully handling exceptions. Think of advice as the specific task your aspect performs at each designated joinpoint.

Types of Advice:

Before: This advice executes immediately before the joinpoint occurs, allowing you to prepare or intercept the operation.
After: This advice executes after the joinpoint has completed, enabling you to react to the outcome or perform cleanup tasks.
Around: This advice grants you complete control over the joinpoint. You can execute code before, during, and after the operation, offering maximum flexibility.

Aspect: This is the glue that brings everything together. It combines pointcuts and advice into a single unit, encapsulating the cross-cutting concern and its behavior. Think of it as a reusable module that adds specific functionality to your application.


Why is Spring AOP this important?

Spring AOP embodies the separation of concerns principle, a fundamental concept of clean and maintainable code. By keeping cross-cutting concerns separate, you improve code readability, reusability, and testability. It's like organizing your workspace, everything has its place, making it easier to find and use what you need. AOP is driven by SRP concept described below.

The Single Responsibility Principle (SRP) is the concept that any single object in object-oriented programing (OOP) should be made for one specific function. SRP is part of SOLID programming principles.
Lastly, here is an exmple of an advice in Spring AOP

    //Intercept method executions of all classes within the com.example.service package
    @Around("execution(* com.example.service.*(..))")
    public Object handle(ProceedingJoinPoint joinPoint) throws Throwable {
      //Capture start time execution
      Instant start = Instant.now();

      try {
          // execute the profiled method
          return joinPoint.proceed();

      } catch (RuntimeException e) {
          throw e;

      } finally {
          long duration = Duration.between(start, Instant.now()).toMillis();

          // Log profiling result
          Signature signature = joinPoint.getSignature();
          LOGGER.info("{},{},{}", signature.getDeclaringType().getSimpleName()
          , signature.getName(), duration);

      }
    }
In this scenario, the Pointcut targets all method executions within the com.example.service package, regardless of the class name and the number of arguments passed. The @Around annotation indicates that the handle() method within the aspect will be invoked around each matching joinpoint. These joinpoints are essentially method calls captured by the Pointcut, and the aspect's logic defined within handle() will be applied before and after each such method execution. The main target of this advice is to measure and log time consumed in order to execute the intercepted method ( JoinPoint )
